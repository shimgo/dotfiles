# コードレビュースキル

プルリクエストの変更内容を7つの観点でレビューし、指摘事項をMarkdownファイルに出力する。

コードの修正はしない。レビューだけを行う。

## 出力形式

- **ファイル個別の指摘**: 指摘対象ファイルと同じディレクトリに `{ベースネーム_rv}.md` を作成する
  - 例: `src/api/handler.go` への指摘 → `src/api/handler_rv.md`
- **全体的な指摘**: リポジトリルートに `./review.md` を作成する
- 指摘事項がない観点はファイルに記載しない
- すべての観点で指摘がなければファイルを作成しない

---

## Step 1: PR情報を取得する

```bash
# 現在のブランチ名を取得
git branch --show-current

# PRの情報を取得（タイトル・本文・ベースブランチ）
gh pr view --json title,body,baseRefName,number,url

# ベースブランチとの差分ファイル一覧
gh pr diff --name-only

# ベースブランチとの差分（全体）
gh pr diff
```

PRが見つからない場合はユーザーにPR番号を尋ねる。

---

## Step 2: 変更内容とその周辺を理解する

差分だけでなく、変更の文脈を理解するために以下を調査する。この調査が不十分だとレビューの質が下がるため、十分な時間をかけること。

1. **変更されたファイルをすべて読む**（差分だけでなくファイル全体）
2. **エントリポイントを特定する**: 変更されたコードがどこから呼び出されるかを辿る
   - HTTPハンドラ、GraphQLリゾルバ、CLIコマンド、バッチ処理など
   - エントリポイントのスキーマ定義（GraphQLスキーマ、OpenAPI定義、protobufなど）も確認する
3. **レスポンスの出口を特定する**: 変更されたコードの結果がどこに返されるかを辿る
4. **周辺の関連コードを読む**: 同じパッケージ・モジュール内の関連ファイル、類似機能の実装
5. **テストファイルを確認する**: 変更対象の既存テスト、同じパッケージの他のテストパターン
6. **DBマイグレーションファイルがある場合**: マイグレーションの内容とそれに依存するコードを確認する

---

## Step 3: 7つの観点でレビューする

以下の7つの観点で変更内容を精査する。各観点で判断に迷ったら「指摘する」側に倒すこと。

### 観点1: 課題と解決方法の整合性

PRの本文に記載されている課題（背景・目的・要件）に対して、実装されている解決策が適切かを確認する。

確認ポイント:
- PRの本文で述べられている課題を実装が正しく解決しているか
- 実装が課題の範囲を超えていないか（スコープクリープ）
- 実装が課題の一部しかカバーしていないということはないか
- 課題の前提条件と実装の前提条件が一致しているか

### 観点2: 代替手法の検討

PRの実装方法以外にどのようなアプローチが考えられるかを検討し、採用されている方法が最善かを評価する。

確認ポイント:
- 2〜3つの代替アプローチを具体的に挙げ、それぞれのメリット・デメリットを比較する
- PRの方法が代替手法より明確に優れていればOK
- 代替手法の方が優れていそうな場合は、PR作者に採用理由の確認が必要である旨を指摘する
- 「なぜこの方法を選んだのか」が自明でない場合は、PRの説明に理由を追記することを推奨する

### 観点3: PRおよび周辺機能の整合性

変更部分だけでなく、その前後の処理フロー全体で一貫性が保たれているかを確認する。

確認ポイント:
- エントリポイント（ルーティング、スキーマ定義）から変更箇所を経由してレスポンスを返すまでの流れが自然か
- 関数名・変数名・コメントが処理の実態と合っているか
- エラーハンドリングが周辺コードと同じパターンに従っているか
- ログメッセージやエラーメッセージが適切か

### 観点4: 用語の統一性

周辺コードとの用語・命名規則の一貫性を確認する。

確認ポイント:
- 同じ概念を指す用語がコードベース内で統一されているか（例: user/account/memberの混在がないか）
- 関数名・変数名の語順が周辺コードと揃っているか（例: `getUserByID` vs `findByIDUser`）
- 新しく導入された用語がある場合、既存の用語との関係が明確か
- コメントやドキュメントでも同じ用語が使われているか

### 観点5: 破壊的な変更の確認

リリース時に安全にデプロイできるかを確認する。特にDBスキーマが先に適用され、その後にサーバーコードがデプロイされるという前提で検証する。

確認ポイント:
- **DBマイグレーション**: 新しいDBスキーマが適用された状態で、古いサーバーコードが正常に動作するか
  - カラム追加: NOT NULL制約にDEFAULT値があるか
  - カラム削除/リネーム: 古いコードがそのカラムを参照していないか
  - テーブル削除: 古いコードがそのテーブルを参照していないか
- **GraphQLスキーマ**: フィールドの削除・リネーム時に、そのフィールドがどこからも参照されていないことを確認する
  - フロントエンドのクエリやフラグメントで使われていないか
  - 他のサービスから呼ばれていないか
- **APIの変更**: レスポンスフィールドの削除や型変更がクライアントに影響しないか

### 観点6: 後方互換性の保証

新しいコードが、変更前から既に存在している古いデータ・古い状態でも安全に動作するかを確認する。観点5が「新スキーマ + 古いコード」のデプロイ安全性を扱うのに対し、こちらは「新コード + 古いデータ」が安全に動くかを扱う。

確認ポイント:
- **新カラム/フィールドのNULL・デフォルト値**: 新しく追加されたカラムやフィールドが、既存レコードでは未設定（NULL・空文字・ゼロ値）になる。新コードのロジックがこの状態を考慮しているか
- **新ロジックの分岐条件**: 新機能の判定条件が、既存データに対して意図しない真偽になっていないか（例: `if user.NewFlag` が古いユーザーで暗黙的に false になり、機能が無効化される／逆に意図せず有効化される）
- **enum/種別の追加**: 新しい区分値が追加された場合、既存レコードの「未分類」「旧区分」を新コードがどう扱うか
- **マイグレーションでのバックフィル**: 古いデータを新スキーマに合わせて埋め直す処理が必要なら、それがマイグレーションや別途スクリプトで提供されているか
- **設定・フィーチャーフラグの初期値**: 新しい設定項目が既存ユーザー・既存テナントでは未設定になる。デフォルト値が安全側に倒れているか
- **永続化されたシリアライズ形式の変更**: JSONカラム・キャッシュ・キュー上のメッセージなど、過去に書き込まれたデータを新コードが読めるか（フィールド追加・削除・型変更）
- **イベント・ジョブのpayload**: キューに既に積まれている古い形式のメッセージを、新しいワーカーが処理できるか
- **APIリクエストの後方互換**: 古いクライアントから送られてくる古い形式のリクエストを、新しいサーバーコードが受け付けられるか（必須化したフィールドが古いクライアントから送られない、など）
- **「過去にこの状態だったレコードが存在しうるか」を必ず問う**: 「現在のコードならこの状態にはならない」だけでは不十分。過去のバグや過去の仕様で生まれた異常データが本番に残っている可能性を考慮すること

### 観点7: テストの実装

PRの変更内容を検証するテストが適切に実装されているかを確認する。

確認ポイント:
- 変更対象の機能に関連する既存テストのパターンを確認し、同じスタイルでテストが書かれているか
- 新しいロジックに対するユニットテストがあるか
- 正常系だけでなく異常系のテストもあるか
- 境界値のテストがあるか
- DBマイグレーションがある場合、マイグレーション前後の動作を検証するテストがあるか
- 既存テストが変更によって壊れていないか

---

## Step 4: 指摘事項をファイルに出力する

### ファイル個別の指摘

指摘対象のファイルと同じディレクトリに `{ベースネーム_rv}.md` を作成する。

```markdown
# コードレビュー: {ファイル名}

PR: #{PR番号} {PRタイトル}

## 観点N: {観点名}

### 指摘1: {指摘の要約}

**該当箇所**: L{開始行}-L{終了行}

```{言語}
// 該当コードを引用
```

**指摘内容**:
{具体的な指摘内容。なぜ問題なのか、どう改善すべきかを記載する}

**重要度**: {高 / 中 / 低}
```

重要度の基準:
- **高**: バグ、データ損失、破壊的変更、セキュリティ問題など、リリース前に必ず対応が必要なもの
- **中**: 一貫性の欠如、テスト不足、代替手法の検討など、対応が望ましいもの
- **低**: 命名の改善提案、コメントの追加など、あると良いもの

### 全体的な指摘

特定のファイルに紐づかない指摘は `./review.md` に出力する。

```markdown
# コードレビュー

PR: #{PR番号} {PRタイトル}
レビュー日: {日付}

## サマリー

{PRの変更内容を1〜2文で要約}

## 全体的な指摘

### 観点N: {観点名}

#### 指摘1: {指摘の要約}

**指摘内容**:
{具体的な指摘内容}

**重要度**: {高 / 中 / 低}
```

---

## Step 5: レビュー結果を報告する

ファイルの出力が完了したら、以下の情報をユーザーに報告する:

- 指摘の総数と重要度別の内訳（高/中/低）
- 出力したファイルの一覧
- 重要度「高」の指摘がある場合はその概要を特に強調して伝える

---

## 注意事項

- コードの修正は行わない。レビュー結果の出力のみを行う。
- PRの本文が不十分で課題が読み取れない場合は、ユーザーに確認する。
- 指摘には必ず具体的な根拠を添える。「〜した方がよい」だけでなく「〜なので〜すべき」と理由を明記する。
- 周辺コードの調査が不十分なまま指摘しない。確信がない場合は調査を深めること。
- `git push` は使用しない。
