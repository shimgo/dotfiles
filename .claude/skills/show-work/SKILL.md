# 作業手順表示スキル

現在のセッションでClaude Codeが実施した作業を、ユーザーがレビューできる形で**画面に**時系列表示する。

このスキルの目的は「透明性の確保」。ユーザーがClaude Codeの作業手順・選択したコマンド・変更内容に問題がないかをその場で確認できるようにする。事後のレビュー用途なので、ファイルには出力しない。

---

## Step 1: セッション全体を振り返る

末尾の数ターンだけでなく、**セッション冒頭から現時点まで**のツール呼び出し履歴を洗い出す。主に以下を拾う:

- 実行したBashコマンド（オプション・引数を含め**原文そのまま**）
- ファイルの新規作成・編集・削除（パスと操作内容）
- スキル呼び出し、サブエージェント起動
- Web検索・フェッチした URL
- 途中で方針変更した箇所、やり直した箇所

推測や要約で補完してはいけない。**実際にツール呼び出しで行った事実のみ**を拾う。未実施のものや「やろうとしたがやらなかった」ものは書かない。

---

## Step 2: 作業をステップに区切る

ツール呼び出し1件ずつを全部並べると冗長になるため、目的単位でステップにまとめる。ただし**コマンドは1件ずつ原文で残す**。例:

- 「状況確認」→ `git status`, `git log --oneline -5`, ファイル2件を Read
- 「修正実施」→ foo.go を Edit, bar.go を Edit
- 「動作確認」→ `go test ./...`

ステップの粒度は「ユーザーがこの単位でOK/NGを判断できるか」を基準にする。

---

## Step 3: 画面に出力する

以下のフォーマットで画面に出力する。コードブロック内のコマンドは実行時の原文をそのまま貼る（省略・整形しない）。

```markdown
## 作業手順

### 1. {ステップの目的を1行で}

実行コマンド:

​```bash
git status
git diff HEAD
​```

参照ファイル:
- `path/to/file.go` を Read

### 2. {次のステップ}

編集内容:
- `path/to/file.go:42` を Edit（関数 `foo` のエラーハンドリング追加）
- `path/to/new_file.go` を新規作成

実行コマンド:

​```bash
go test ./path/to/...
​```

結果: 通過

### 3. {最後のステップ}
...
```

---

## 書き方のルール

- **コマンドは原文そのまま**: `git status` と書いたなら `git status` のまま。`git status -s` だったら `-s` も残す。勝手に整形したり省略しない。ヒアドキュメントや長いワンライナーも同様。
- **操作内容は簡潔に**: 「〜のため」の理由は不要。何をしたかだけ書く（理由はユーザーが会話履歴から判断する）。
- **失敗・やり直しを隠さない**: 途中で失敗したコマンド、差し戻されて修正したケースも残す。隠すとレビューの意味がなくなる。
- **実施していないものは書かない**: 「〜する予定だった」「〜を検討した」は書かない。実際にツール呼び出しした事実のみ。
- **ファイル出力はしない**: このスキルは画面出力専用。`letter.md` のようなファイルは作らない。

---

## 注意事項

- セッションが長大でも末尾だけを要約しない。冒頭のセットアップや途中の方針変更もレビュー対象になる。
- TodoWrite でタスクを管理していた場合、それ自体は作業ではなく進行管理なので列挙しなくてよい（ただしタスク内容が作業の区切りとして有用なら使ってよい）。
- ユーザーが特定の範囲（例: 直近10分、特定ファイルに関する作業のみ）を指定した場合はその範囲に絞る。
