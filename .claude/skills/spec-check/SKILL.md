# 仕様確認・差分提示スキル

コードの実装・テスト・コメントから**現在の仕様**を読み取り、ユーザーが認識している仕様と比較して差分を提示する。

## このスキルが扱う2つのモード

### モード1: 仕様の説明
ユーザーが「〇〇の仕様を教えて」「この機能はどう動くの？」と聞いてくる場合。
→ コードを調査して現在の仕様を整理して説明する。

### モード2: 仕様の差分確認
ユーザーが「〇〇はXXという動作をするはず」「自分の認識と合っているか確認して」のように
**自分の想定を持って**確認を求めてくる場合。
→ ユーザーの想定と実際の実装を比較し、差分を提示する。

どちらのモードかを会話から判断する。両方含む場合は両方対応する。

## 調査手順

### Step 1: 対象を特定する

ユーザーの入力（テキスト、ファイルパス、関数名など）から調査対象を特定する。

- **テキスト形式**（例: 「注文確定の処理」）→ 関連キーワードで Grep してファイル・関数を特定
- **ファイルパス形式**（例: `./src/order.go`）→ 直接 Read して調査
- **関数名形式**（例: `ConfirmOrder`）→ Grep で定義箇所を特定

対象が曖昧な場合は、候補をユーザーに提示して確認を取ってから進む。

### Step 2: 現在の仕様を読み取る

以下の優先順位でコードを読み、現在の仕様を導き出す。

1. **テストファイル**（最優先）
   - テストケース名・入力値・期待値から仕様の境界条件を読み取る
   - テーブル駆動テストがあれば、各ケースが仕様の一つ一つを表している
   - `_test.go`, `*.test.ts`, `*_spec.rb` などを Grep で探す
   - 調査対象フィールドと同じ出力構造に含まれる関連フィールドを特定し、テストケースの具体的な期待値からそれらの関係性（包含・排他・合算など）を確認する

2. **実装コード本体**
   - 条件分岐・バリデーション・計算ロジックから動作を読み取る
   - エラーケースの扱いも仕様の一部として読み取る

3. **コードコメント・ドキュメントコメント**（補助的に使用）
   - コメントは古くなっている可能性があるため、実装コードから読み取った仕様を**補強する目的**で参照する
   - コメントをそのまま仕様として断定しない。必ず実装コードと照合して確認する
   - コメントが他の箇所（関連ファイル・関数など）に言及している場合は、その箇所を実際に読みに行く手がかりとして活用する

### Step 3: 仕様を整理する

読み取った内容を以下の観点で整理する。

- **正常系**: どういうインプットに対してどういう結果を返すか
- **異常系・エラーケース**: どういう場合にエラーや例外になるか
- **境界条件**: 値の上限・下限、空値・null の扱いなど
- **副作用**: DBへの書き込み、外部サービスへの呼び出しなど

### Step 4: 差分を比較・提示する（モード2のみ）

ユーザーの想定と実装を照合し、差分を列挙する。

差分がある場合は以下の形式で提示する：

```
## 仕様の差分

### 差分1: [差分の概要]

ユーザーが想定する実装: [ユーザーの認識]
現在の実装: [実際の動作。根拠となるファイルパス:行番号を付記する]

### 差分2: [差分の概要]

ユーザーが想定する実装: [ユーザーの認識]
現在の実装: [実際の動作。根拠となるファイルパス:行番号を付記する]
```

差分がない場合：
```
## 仕様の差分

差分はありません。ユーザーの認識と実装は一致しています。

## 現在の仕様: [対象の名前]

### 正常系
- ...

### 異常系・エラーケース
- ...
```

## 出力フォーマット

### モード1（仕様の説明）の出力形式

```
## 現在の仕様: [対象の名前]

### 正常系
- [動作の説明] （根拠: ファイルパス:行番号）
- ...

### 異常系・エラーケース
- [条件]: [結果] （根拠: ファイルパス:行番号）
- ...

### 境界条件
- [条件]: [結果] （根拠: ファイルパス:行番号）
- ...
```

### モード2（差分確認）の出力形式

まず現在の仕様の概要を示し、続けて差分を提示する。

```
## 現在の仕様: [対象の名前]

[仕様の簡単な説明]

---

## 仕様の差分

### 差分1: ...
ユーザーが想定する実装: ...
現在の実装: ...
```

## 注意事項

- **調査範囲**: 特に指定がない限り、プロジェクト全体（すべてのファイル・ディレクトリ）を対象に調査する
- **根拠の明示**: 「現在の実装」の根拠となるファイルパスと行番号を必ず付記する。省略しない
- 根拠が不明な推測で仕様を断言しない。コードを読んで確認できた内容のみを「現在の仕様」として提示する
- 仕様が読み取れなかった箇所は「コードから確認できませんでした」と正直に伝える
- テストが存在しない場合はその旨を伝え、実装コードのみから読み取った仕様であることを明示する
- **外部システムのAPIドキュメントや自動生成コードのコメントは補助情報に留める**。「何が含まれるか」はテストケースの入力・期待値から直接確認する
- 調査対象フィールドが「含む/含まない」という仕様の確認依頼の場合、関連フィールドとの関係をテストで検証するまで結論を出さない
