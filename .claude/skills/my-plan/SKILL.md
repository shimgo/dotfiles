# 実装計画作成スキル

`./task.md`に記載された仕様を読み取り、コードベースを調査した上で、テスト駆動開発に基づく実装計画を`./plan.md`に出力する。

実装はしない。計画を立てるだけ。

## 手順

### Step 1: 仕様の理解

`./task.md` を Read で読み、実装する機能の仕様を理解する。

- 機能の目的、入出力、ビジネスルールを整理する
- 以下の観点で仕様に問題がないか検証する
  - **矛盾**: 仕様内で相反する記述がないか
  - **曖昧さ**: 複数の解釈ができる記述がないか
  - **欠落**: 境界条件やエラーケースの記載が抜けていないか
- 問題が見つかった場合は、**先に進まず**、問題点をユーザーに伝えて判断を仰ぐ
- 不明点があればこの段階でユーザーに質問する
- **この段階では実装しない**

問題が見つかった場合の出力形式:

```
## 仕様の確認事項

### 問題1: [概要]
該当箇所: task.md の「〇〇」の記述
問題の内容: [具体的な矛盾・曖昧さの説明]
考えられる解釈:
  - A: ...
  - B: ...
→ どちらの解釈が正しいですか？
```

問題がなければ次のステップへ進む。

### Step 2: コードベースの理解

Step 1 で把握した仕様を踏まえ、関連するコードベースを調査する。

1. **データベーススキーマ**
   - `schema.sql` ファイルを Glob で探し、Read で内容を確認する
   - テーブル構造、リレーション、制約を把握する
   - 仕様に関連するテーブルを重点的に読む

2. **GraphQLスキーマ**
   - `resolver/` 配下の `*.graphql` ファイルを Glob で探し、Read で内容を確認する
   - 既存のQuery、Mutation、型定義を把握する
   - 仕様に関連するスキーマを重点的に読む

### Step 3: 既存の類似機能との整合性検証

`./task.md` の仕様に関連する既存の類似機能を調査し、整合性・統一性を確認する。

- 既存の類似機能の実装パターン（domain構造体、usecase関数、GraphQLスキーマの命名規則など）を確認する
- 新機能の仕様が既存機能と矛盾していないか、命名規則や設計パターンに統一性があるかを検証する
- 既存機能で確立されたパターンがあれば、それに倣う方針とする

問題が見つかった場合は、問題点をユーザーに伝えて判断を仰ぐ。

```
## 既存機能との整合性に関する確認事項

### 問題1: [概要]
既存機能: [機能名、ファイルパス:行番号]
新機能の仕様: [task.md の記述]
不整合の内容: [具体的な説明]
→ 既存機能に合わせますか？それとも新しい方針にしますか？
```

問題がなければ次のステップへ進む。

### Step 4: 実装要素の決定

Step 1〜3 の調査結果を踏まえ、以下を決定する。

1. **テーブル変更**: 新規テーブル作成、既存テーブルへのカラム追加、インデックス追加など
2. **domain構造体名**: 既存の命名規則に従って決定する
3. **usecase関数名**: 既存の命名規則に従って決定する

### Step 5: `./plan.md` への出力

以下のフォーマットで `./plan.md` に実装計画を出力する。

```markdown
# 機能概要
実装する機能の概要を記載する。

# 参照するテーブル
機能の処理で参照するテーブル名やフィールド名を記載する。

# 新しく作成するdomainの構造体
新しく作成する予定のdomainの構造体とその説明を記載する。

# DBスキーマの変更内容
DBスキーマの変更内容を記載する。スキーマの変更が不要な場合は、その旨を記載する。

# GraphQLスキーマの変更内容
GraphQLスキーマの変更内容を記載する。スキーマの変更が不要な場合は、その旨を記載する。

# E2Eテストの変更内容
E2Eテストにどんな変更を加えるか、どんな操作と検証を行うかを記載する。
E2Eテストシナリオを新規作成する場合の例:
  以下のE2Eテストを追加する。
  - {テスト関数名1}: {./path/to/test1}
    1. Orderを作成する
    2. Orderをキャンセルする
    3. OrderのSubtotalが0になっていることを確認する
  - {テスト関数名2}: {./path/to/test2}
    1. Orderを作成する
    2. Orderを一部返金する
    3. Orderをキャンセルする
    4. OrderのSubtotalが0になっていることを確認する
E2Eテストシナリオを修正する場合の例:
  以下のE2Eテストを修正する。
  - {テスト関数名1}: {./path/to/test1}
    - 現在のテスト内容
      1. Orderを作成する
      2. Orderをキャンセルする
      3. OrderのSubtotalが0になっていることを確認する
    - 変更内容
      a. 3でOrderのTotalPriceが0になっていることを確認する手順を追加する

# 実装手順
実装の手順をチェックリストにして記載する。
テスト駆動開発の手順に従うこと（下記「実装手順のルール」を参照）。

# 参考にしたファイル
参考にしたファイルや類似のコードの箇所などが見つかったら、そのファイル名、行数、関数名などを記載する。
```

必要に応じてセクションを追加してもよい。

## 実装手順のルール

実装手順は**TDD（テスト駆動開発）**の原則に従う。これにより、実装前にテストで期待する動作を定義し、テストが失敗することを確認してから実装に着手できる。

### 新規機能の場合

1. GraphQLスキーマの修正（必要な場合）
2. ロジックを持たない最低限のエンドポイント（リゾルバ）を作成
3. E2Eテストを実装（この時点では失敗する）
4. **ユーザーからE2Eテストが問題ないことの承認を得る**
5. domain、usecase等を実装してE2Eテストを通す

### 既存機能の修正の場合

1. E2Eテストを変更（この時点では失敗する）
2. **ユーザーからE2Eテストが問題ないことの承認を得る**
3. 実装を変更してE2Eテストを通す

### 共通ルール

- E2Eテストが失敗することを確認したら、**必ずユーザーに確認して承認を得る**ステップを入れる
- E2Eテストの作成・修正が不要と判断した場合は、ユーザーの承認を得る必要はなく、ユニットテストで同様にTDDの手順を踏む
- 承認を得てから実装に移る手順にする
- チェックリスト形式（`- [ ]`）で記載する

## 注意事項

- このスキルは**計画を立てるだけ**であり、コードの実装は行わない
- `./task.md` が存在しない場合はユーザーに伝える
- Step 3、Step 4 で問題が見つかった場合は、ユーザーの判断を待ってから先に進む。勝手に解釈して計画を立てない
- plan.md の出力後、ユーザーに内容を確認してもらい、修正が必要であれば対応する
