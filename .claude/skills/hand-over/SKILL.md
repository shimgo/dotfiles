# 引き継ぎ資料作成スキル

現在のセッションで行った対話・調査・編集内容を、次のセッションや別の担当者が拾い直せる形にまとめて `./letter.md` に出力する。

出力先は **Claude Codeを起動したディレクトリ（カレントディレクトリ）直下の `./letter.md`** で固定する。

---

## Step 1: 現状を把握する

会話履歴だけでなく、作業環境の最新状態も引き継ぎ資料に含める必要があるため、以下のコマンドで現状を確認する。

```bash
git status
git branch --show-current
git log --oneline -10
```

リポジトリ外で作業している場合や、関係ない場合はスキップしてよい。

---

## Step 2: セッション全体を振り返る

末尾の数ターンだけでなく、**セッション全体**を振り返る。序盤に決まった方針や前提が埋もれやすい。以下を洗い出す:

- ユーザーの最初の依頼と、途中で設定された目標
- 実際にツール呼び出しで行った作業（推測ではなく事実）
- 途中で止まった作業、未対応のままの指摘
- ユーザーから受けたフィードバック、好み、制約
- 失敗した試みや詰まったポイント

---

## Step 3: letter.md を書き出す

以下の構成で `./letter.md` を作成する。該当情報が無いセクションは省略してよい。

```markdown
# 引き継ぎメモ

作成日: {YYYY-MM-DD}
ブランチ: {現在のブランチ名}

## 概要

{このセッションの目的を1〜2文で。ユーザーの最初の依頼、もしくは途中で設定された目標}

## やったこと

- {時系列で作業内容を箇条書き。成果物はファイルパス付きで}
- {実際にツール呼び出しで行った作業のみを書く。推測は書かない}

## 現状

- {git status, 編集中でコミットしていないファイル}
- {ブランチ、PRの状態}
- {実行したコマンドと結果}

## 次にやること

1. {未完了タスクを優先度順に列挙}
2. {会話内で "TODO" と明示されたもの、途中で止まった作業、ユーザーが最後に言及したが未対応のものを必ず拾う}

## 重要な前提・決定事項

- {作業を進める上で必要な背景知識、制約、合意事項}
- {「なぜそうしたか」の判断根拠。コードやコミットログから読み取りにくいもの}
- {ユーザーから指摘された好みや注意事項}

## ハマりどころ・注意点

- {今回詰まった点、間違えやすい点、失敗した試み}
- {同じ罠に次の担当者が落ちないための注意}

## 参考情報

- {参照したファイル: `path/to/file.go:123` のように行番号付きで}
- {URL、コマンド、外部ドキュメント}
```

---

## 書き方のルール

- **途中で止まった作業を隠さない**: 未完了は必ず「次にやること」に書く
- **即参照できる形式**: ファイルパスは `path/to/file.go:123` のように行番号付き、コマンドはコピペで実行できる形で書く

---

## 注意事項

- **Claude Codeを起動したディレクトリの外にファイルを書き出さない**。出力先は `./letter.md` のみ
- `letter.md` が既に存在しても確認せず上書きする
- 長大なセッションでも全体を振り返る。末尾だけを要約しない
